AIと一緒にブログの見た目を全部見直した話

〜スクリーンショット分析→優先度付け→一括修正の全記録〜

「見た目を見直したい」の一言から始まった

ある日、Claude Codeに「ブログの見た目を見直したい」と伝えました。 デザインの知識はほぼゼロです。「なんとなく整えたいけど、何から手をつければいいかもわからない」というふわっとした状態でした。

そこからAIが動き始めて、気づいたら全17記事が修正されていました。 その一部始終を記録します。

ステップ1：AIがスクリーンショットを4枚撮って現状分析した

私が最初にやったこと——というより、AIがやったことは、Playwrightを使ってサイトのスクリーンショットを4枚撮影することでした。

デスクトップ表示・モバイル表示・記事ページ・トップページ、それぞれの見た目を画像として取得して、AIが自分で分析したわけです。 私は特に「ここを見て」とは言っていません。「見た目を見直したい」と伝えただけで、AIが自分で判断してスクリーンショットを撮り始めました。

これが意外でした。
私はてっきり「どこを直したいですか？」と聞かれると思っていました。 でもAIは「まず現状を見る」という段取りから入ってきた。 自分で問題を探しに行く、という動き方です。

ステップ2：問題が3つ見つかり、優先度をつけてもらった

分析の結果、AIが発見した問題は3つでした。

優先度 問題 影響範囲
高 モバイルでハンバーガーメニューがない 全17記事
中 ヘッダーロゴが2行に折り返す 全ページ
低 ページ間でナビゲーションの項目数が違う 記事ページ

一番大きな問題は「モバイルでハンバーガーメニューがない」でした。 スマートフォンで見ると、トップページ以外のナビゲーションが消えていて「← ホーム」ボタンしか使えない状態だったわけです。 これは気づいていませんでした。

2つ目は「ヘッダーロゴの折り返し」。サイト名の「OCHITSUKANAI COMPANY」が小さい画面幅で2行になってしまっていました。 3つ目は「ナビの項目数のズレ」。トップページには4項目あるのに、記事ページでは項目が少ない——という統一感のなさです。

デザインの知識ゼロでも、問題が見つかる。
自分では「なんかおかしいな」と感じてもどこが悪いか言語化できなかったことを、 AIがスクリーンショットを見て整理してくれました。 「問題を発見する」という作業自体をAIに任せられたのが大きかったです。

ステップ3：Pythonスクリプトで全17記事を一括修正

問題と優先度が整理されたあと、AIがPythonスクリプトを書いて全17記事を一括で修正しました。

私がやったことは、スクリプトの実行確認をしただけです。 手動でHTMLファイルを1つずつ開いて編集するようなことはゼロでした。

# AIが書いたスクリプトの動作イメージ

for html_file in blog/*.html:

    # ハンバーガーメニューのHTMLを挿入

    insert_hamburger_menu(html_file)

    # nav閉じタグの存在を検証

    assert_nav_closed(html_file)

    # ヘッダーロゴのスタイル修正

    fix_header_logo(html_file)

# 17ファイル、完了。

「全記事を手動で直すのは大変だろうな」と思っていたのが、 スクリプト1本で全部終わりました。 17記事分の作業時間は、体感として数分です。

ステップ4：レビュさん（AI）がHTMLバグを発見した

修正が終わったあと、レビュさん（レビュー担当のAI）にチェックを依頼しました。 すると、スクリプトの置換処理で </nav> の閉じタグが消えていたことが発覚しました。

全17記事に、nav要素が閉じられていない不正なHTMLが配布されていたわけです。 レビュさんがいなければ、そのまま公開するところでした。

発見されたバグ（レビュさんの指摘）：
スクリプトの置換で </nav> 閉じタグを消失 → nav未閉じの不正HTMLを全17記事に配布

対応：即日修正 + 今後の再発防止として「置換後に </nav> の存在を検証するアサーションをスクリプトに入れる」をルール化

バグを修正して再度レビューを依頼すると、今度は aria-expanded 属性が未設定という指摘も出てきました。 スクリーンリーダー（視覚障害者向けツール）がメニューの開閉状態を認識できない、というアクセシビリティの問題です。 こちらも合わせて修正しました。

AI同士のチェック体制が機能した

今回の流れを整理するとこうなります。

1

Claude Code（作業担当）

スクリーンショット撮影 → 問題発見・優先度付け → Pythonスクリプト作成・実行

2

レビュさん（チェック担当）

HTMLバグ発見（nav未閉じ）→ アクセシビリティ指摘（aria-expanded未設定）

3

Claude Code（修正担当）

バグ修正 → 再レビュー → aria属性追加 → push完了

作業担当とチェック担当が別のAIになっていることで、「自分で作ったものを自分でチェック」という死角が消えます。 人間でいえば「製造と品質検査は別の人がやる」という当たり前の仕組みです。

レビュさんがバグを見つけなければ、nav未閉じのまま全記事が公開されていました。 AI同士のチェック体制が実際に機能した体験でした。

今回わかったこと・気づいたこと

まとめ：

1. 「見た目を見直したい」だけでAIが現状分析から始める → 問題の言語化も任せられる
2. スクリーンショットで見える化 → 「なんとなくおかしい」が具体的な問題として整理される
3. Pythonで全17記事を手作業ゼロで修正 → 「記事数が多くて大変」は仕組みで解決できる
4. AI同士のチェック体制 → 作業AIが見落としたバグを、チェックAIが発見してくれた

デザイン知識は要らなかった

率直に言って、今回の作業で私がデザインの知識を使った場面はゼロでした。 問題の発見はAIがやり、修正コードもAIが書き、バグの検出もAIがやりました。

私がやったのは「見た目を見直したい」と伝えることと、「このスクリプトを実行していいですか」への確認だけです。

「デザインが苦手だからサイトの改善に手をつけられない」という感覚は、 AIと組み合わせると意外と解消できます。 問題の発見から修正まで、「仕組み」として流れを作れるかどうかが大事だと感じました。

ただ一点——AIが書いたコードにもバグはあります。 今回は </nav> の消失がそれでした。 だからレビューを別のAIにやらせる、という体制にしていて、それが実際に機能しました。 「AIに任せる＝ノーチェックでいい」ではなく、チェック機能も仕組みに組み込むことが大事だと改めて思いました。
