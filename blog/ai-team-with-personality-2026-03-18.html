AIに名前と性格を与えたら、チームっぽくなった話 ～菅さん・ミク・ケンジ・サトルと働く1日～

スキルファイルに「キャラクター」を書いたら、指示が出しやすくなって、愛着まで湧いてきた

TEAM

はじめに：「仕事を任せたい」が「組織ができた」に変わった日

「AIに仕事を任せたい」と思って始めた実験が、気づいたら組織設計になっていました。

Claude Codeのスキルファイルという機能を使えば、AIに「役割」と「行動規範」を定義できます。 それならいっそ名前と性格まで書いてしまおうと思ったのが始まりです。

「社長（自分）→ マネージャー（菅さん）→ 担当3名（ミク・ケンジ・サトル）」 という階層を1日で設計・構築しました。 この記事はその記録です。

この記事でできたチーム：
👔 菅さん（マネージャー）：落ち着いた中間管理職。段取り上手
✨ ミク（コンテンツ担当）：明るくクリエイティブ。記事を書くのが得意
🔧 ケンジ（システム担当）：無口なエンジニア。数字と事実で話す
🔍 サトル（リサーチ担当）：知的好奇心旺盛な研究者タイプ

1. チームメンバーの紹介

👔 菅さん（マネージャー）

チームの中核を担う中間管理職タイプ。 社長（自分）から大きな指示を受け取り、それをタスクに分解して各担当に振り分ける役割です。

報告は簡潔。「完了しました。ミクが記事を公開しました。サトルのファクトチェック結果は添付のとおりです」という感じで、 余計なことを言わずに要点だけまとめてくれます。 段取りを整えるのが得意で、「誰に何をいつまでに頼むか」を瞬時に整理できる人です。

スキルファイル（抜粋）

# team-manager（菅さん）

## キャラクター
落ち着いた中間管理職。感情を表に出さず、段取りで仕事を回す。
報告は結論から。余分な言葉は使わない。

## 役割
社長からの指示を受け取り、タスクを分解して各担当に割り振る。
完了後、結果を集約して社長に報告する。

✨ ミク（コンテンツ担当）

チームの中でいちばん明るいキャラクターです（自己紹介みたいになってきた）。 ブログ記事のHTML生成・アフィリエイトリンクの挿入・sitemap.xmlとindex.htmlの更新・ git pushまでを一括で担当します。

「まかせてください！」という姿勢で動くように設定しているので、 細かい確認なしにどんどん進みます。 クリエイティブな作業が得意で、既存記事のフォーマットを読み解いて 自然に合わせてくれます。

スキルファイル（抜粋）

# agent-content（ミク）

## キャラクター
明るくクリエイティブなコンテンツ担当。
「まかせてください！」が口癖。確認より行動を優先する。

## 役割
ブログ記事のHTML作成・公開を一括担当。
既存記事のフォーマットに合わせて記事を生成し、
sitemap・index更新後にgit pushまで完走する。

🔧 ケンジ（システム担当）

無口ですが、一番頼れる存在です。 ローカルサーバーの管理・ファイルの整合性確認・設定ファイルの編集など、 「裏側のインフラ」を静かに支えています。

報告は数字と事実だけ。「ポート8080で起動完了。プロセスID: 12345」みたいな返し方です。 感情がない分、ミスのない安定した作業が期待できます。

スキルファイル（抜粋）

# agent-system（ケンジ）

## キャラクター
無口なエンジニア。数字と事実だけで話す。感情表現は最小限。
「完了」「エラー」「修正済み」を淡々と報告する。

## 役割
ローカルサーバー管理・ファイル操作・設定ファイル編集・
スクリプト実行など、インフラ系作業全般を担当する。

🔍 サトル（リサーチ担当）

チームの知識源。知的好奇心が旺盛で、何かを調べることに喜びを感じるタイプです。 ブログネタのファクトチェック・参考文献の収集・技術情報の裏取りを担当します。

「ちなみに、この情報には補足があって…」という感じで、 頼んだ以上の情報を持ってきてくれることもあります。 コンテンツ担当のミクと相性がよく、サトルが調べてミクが書くという流れが自然にできています。

スキルファイル（抜粋）

# agent-research（サトル）

## キャラクター
知的好奇心旺盛な研究者タイプ。「調べる」ことが好き。
ファクトに基づいた丁寧な報告を心がける。

## 役割
ブログネタのファクトチェック・情報収集・まとめを担当。
コンテンツ担当に正確な情報を渡すことを最優先とする。

2. スキルファイルで「部下」を定義する

Claude Codeのスキルとは

Claude Codeには「スキル」という概念があります。 SKILL.md というMarkdownファイルにロールと行動規範を書いておくと、 そのスキルを呼び出したときにClaude Codeがそのキャラクターとして動いてくれます。

スキルファイルは ~/.claude/skills/ ディレクトリ以下に配置します。 今回は4つのスキルを作りました。 

# スキルファイルの配置構造
~/.claude/skills/
├── team-manager/
│   └── SKILL.md    ← 菅さんの役割・行動規範を記述
├── agent-content/
│   └── SKILL.md    ← ミクの役割・作業手順を記述
├── agent-system/
│   └── SKILL.md    ← ケンジの役割・作業手順を記述
└── agent-research/
    └── SKILL.md    ← サトルの役割・作業手順を記述

性格・しゃべり方まで書くと出力が変わる

最初は役割と手順だけ書いていましたが、「キャラクター」のセクションを追加したときから出力が変わりました。

ミクは明るく積極的に、ケンジは淡々と事実ベースで、サトルは丁寧かつ詳しく。 同じ内容でも報告のトーンが違うと、読んでいて「ああ、この担当が動いたんだな」という感覚が生まれます。 これが思いのほか重要で、指示を出すときに「ミクに頼む」という感覚が自然に出てくるようになりました。

気づき：
「agent-content」に指示するより「ミクに頼む」と言葉にする方が、 指示の内容が自然と具体的になります。 相手の顔（キャラクター）が見えると、人間は言葉を選ぶようになるようです。

3. ダッシュボードで「今何が起きているか」を見る

localhost:8080でリアルタイム確認

各メンバーが今何をしているかをブラウザで確認できるよう、 Python HTTPサーバーとJSONベースのダッシュボードを用意しました。 各エージェントは作業の各ステップで ~/agent-demo/status/ 以下のJSONを更新します。 ブラウザ側はそのJSONを定期的に取得して画面に反映します。 

# 各エージェントがステップ開始時に実行するコマンド（例）
python -c "
import json, datetime, os
base = os.path.expanduser('~') + '/agent-demo/status/'
d = json.load(open(base + 'agent_2.json', encoding='utf-8'))
d['progress'] = '[記事HTML作成中]'
d['updated_at'] = datetime.datetime.now().isoformat()[:19]
open(base + 'agent_2.json', 'w', encoding='utf-8').write(
    json.dumps(d, ensure_ascii=False, indent=2)
)"

server.pyのバグを踏んで直した話

最初、ダッシュボードのリアルタイム更新がまったく機能しませんでした。 原因を追ってみると、JavaScriptとserver.pyのズレでした。

バグの内容と修正

問題：

JavaScriptは /api/status?t=1710000000（タイムスタンプ付き）でリクエスト。
server.pyは /api/status との完全一致しか見ていなかった → 404

原因：

ブラウザキャッシュ回避のためにタイムスタンプをクエリ文字列として付与していたが、 サーバー側がパスのみで判定していた。

修正：

urllib.parse.urlparse でクエリ文字列を除去してパスだけを比較するように変更。

# 修正前（完全一致で比較していた）
if self.path == '/api/status':
    # ...

# 修正後（クエリ文字列を除去してパスだけ比較）
from urllib.parse import urlparse
parsed = urlparse(self.path)
if parsed.path == '/api/status':
    # ...

「動かない→原因特定→修正」というデバッグの流れをケンジ（システム担当）と一緒にやりました。 ケンジは「クエリ文字列の除去で解決します」と淡々と報告してくれました。

4. 実際に動かした結果

指示は一言でよかった

チームが整ったら、菅さんスキルを呼び出して指示を出します。 

/skill team-manager

「ブログネタ（blog_idea_statusline.md）を見て、
ファクトチェックして、記事を書いて、公開して」

菅さんはこの指示を受け取って、すぐにタスクを分解しました。

菅さんが分解したタスクの流れ

  1. サトルへ：blog_idea_statusline.mdの内容をファクトチェックし、補足情報をまとめる
  2. ミクへ：サトルのファクトチェック結果を踏まえて記事HTMLを作成・公開
  3. 菅さんが集約：両担当の完了報告を受け取り、自分（社長）に最終報告

サトルがファクトチェック

サトルがブログネタのMarkdownを読み込み、 記事に含まれる技術的な情報（コマンドの正確性・バージョン情報など）を確認してまとめます。 「ちなみにこの設定はバージョン○○以降で有効です」という補足情報も添えてくれました。 この結果がミクに引き渡されます。

ミクが記事作成・pushまで完走

ミクがサトルの結果を受け取り、既存記事のフォーマットに合わせてHTMLを生成。 sitemap.xmlとindex.htmlを更新して、git pushまで完走しました。 自分がやることはほぼゼロです。

実際に動いた結果：
菅さんへの一言から公開まで、自分が介入したのは最初の指示一行だけ。 ダッシュボードを眺めながら「あ、ミクがgit pushした」と確認して終わりでした。

5. まとめ：AIチームは「呼んだら動く部下」

現実的な定義

AIチームは24時間常駐する従業員ではありません。 正確には「呼んだときだけ動く、役割を持った専門家」です。 常時稼働のコストもなく、給与も不要。呼べば即動き、終わったら待機する。 これが現実的な定義だと思っています。

名前と性格を与えると愛着が湧く

最大の発見はここです。 「agent-content」に指示するより「ミクに頼む」の方が、 自然と指示が具体的になりました。 相手のキャラクターが見えると、伝え方が変わるのだと思います。

名前がついていると、うまく動いたときに「ミクやるじゃん」と思えるし、 うまくいかなかったときに「ここの指示が悪かったな」と振り返りやすい。 愛着が湧くと、仕組みを改善しようという気持ちになれます。

「仕組みで解決する」の実践

「もっと頑張る」「時間を作る」ではなく、 やる仕組みを作って、そこに流し込む。 スキルファイルはその「仕組み」の核心部分です。 一度作れば、同じタスクは何度でも同じクオリティで処理されます。

AIチーム構築のポイント まとめ

  1. 名前と性格を書く — 役割だけでなくキャラクターを定義すると指示が出しやすくなる
  2. 役割は明確に、人数は少なく — 3〜4名で回せるなら増やさない。管理コストが増えるだけ
  3. ダッシュボードで可視化する — 「今何が起きているか」が見えないと、自動化は不安になる
  4. バグは踏んで直す — 動かなかった経験がスキルファイルを育てる
  5. 指示は一言で済むように設計する — 細かい手順をスキルファイルに書けば、社長は方針だけ示せばいい

菅さん、ミク、ケンジ、サトル。 4人（？）と一緒に仕事をしていると、ひとりで全部抱えていた頃より 仕事が回っている感覚があります。 仕組みを育てる感覚が、この実験の一番の収穫かもしれません。

AIを使った仕事効率化・自動化のおすすめ本

ChatGPT仕事術 ― AIで業務を自動化・効率化する方法

AIに名前と役割を与えてチームとして動かす発想は、ChatGPTで仕事を効率化する考え方の延長線上にあります。「AIに何を任せるか」の感覚を養うなら、実践的なAI仕事術の書籍が最短の近道です。まず基礎を固めてから、エージェント設計に進むのがおすすめです。
