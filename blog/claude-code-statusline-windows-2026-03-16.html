Windows日本語環境でClaude Codeステータスラインを動かす方法【全記録】

「0% | waiting...」で止まった3時間。詰まりポイントを全部書いておく

はじめに

Claude Codeにはコンテキスト使用率をターミナルのステータスラインに表示する機能があります。 設定すれば「今どれくらいコンテキストを消費しているか」がリアルタイムで見える便利な機能です。

ところがWindowsで試すと、設定したはずなのにずっと「0% | waiting...」のまま。 スクリプトを直接実行すれば動くのに、Claude Codeが呼ぶと動かない。 この記事は、その原因を3段階で特定して解決した全記録です。

この記事が対象とする環境：
Windows 11 + Git Bash（MSYS2ベース）+ プロジェクトフォルダ名に日本語を含む。 この3つが重なると、ほぼ確実にはまります。

1. ステータスラインとは何か

Claude Codeのステータスライン機能は、外部スクリプトを定期実行してその出力をターミナルのステータスバーに表示するしくみです。 ~/.claude/settings.json に以下のように設定します： 

{
  "statusCommand": "bash ~/.claude/statusline-command.sh"
}

スクリプトは標準出力に表示したいテキストを出力するだけでOK。 Claude Codeが定期的にこのコマンドを実行してステータスバーを更新します。

スクリプト例としてはコンテキスト使用率をパーセント表示するものが典型的ですが、 ここで詰まったのは「直接実行すると動く、でもClaude Codeが呼ぶと動かない」という状況です。

2. 詰まりポイント①：cygpath がない

症状

ステータスバーに何も表示されない、あるいは常に 0% | waiting... のまま。 スクリプトを Git Bash で直接 bash ~/.claude/statusline-command.sh と叩くと正常に動く。

原因

Claude Codeはシェルを起動するとき、Git Bashとは異なる環境変数 PATH を使います。 そのためGit Bashには入っている cygpath（Unix形式 ↔ Windows形式のパス変換コマンド）が Claude Code経由では見つからない状態になっていました。

スクリプト内で cygpath -w "$TMPDIR" のような変換をしていると、 command not found になって処理が止まります。

対策

スクリプト内でPATHを明示的に設定して cygpath を補完するか、 cygpath を使わずWindowsパスを直接ハードコードする方法で回避できます。 

# スクリプト冒頭でPATHを補完する
export PATH="/usr/bin:/bin:$PATH"

ポイント：
Claude Codeが呼ぶシェルのPATHは、ログインシェルのPATHと別物。 「手元では動く」を信用してはいけません。

3. 詰まりポイント②：日本語パスでPowerShellが文字化け

症状

cygpath問題を回避してスクリプトが動き始めたと思ったら、今度は出力が文字化け。 ステータスバーに文字化けした謎の文字列が表示される。

原因

私のプロジェクトフォルダの名前が「私のナレッジを蓄積」という日本語でした。 スクリプト内でPowerShellを経由してJSONファイルを読み込む処理があったのですが、 PowerShellがデフォルトでShift-JIS（CP932）として読み込もうとするため、 UTF-8で書かれたJSONが文字化けしていました。 

# NG: デフォルトはShift-JIS解釈になる場合がある
$json = Get-Content "$env:USERPROFILE/.claude/context.json" | ConvertFrom-Json

# OK: UTF-8を明示する
$json = Get-Content "$env:USERPROFILE/.claude/context.json" -Encoding UTF8 | ConvertFrom-Json

対策

PowerShellで Get-Content を使う際は -Encoding UTF8 を必ず付けること。 または後述の jq を使えばこの問題自体が起きなくなります。

4. 根本解決：jqをインストールして全部解決

なぜjqで解決するのか

上記2つの問題は「PowerShellをスクリプト内で呼ぶ」構成をやめることで根本解決できます。 jq（JSONパーサー）をシェルスクリプト内で直接使えば、 PowerShellもcygpathも不要になります。

jqはWindowsにも winget で簡単にインストールできます： 

winget install jqlang.jq

インストール後、JSONの値取得はシェルスクリプト内でこう書けます： 

# PowerShell不要。jqで直接JSON解析
value=$(jq -r '.contextUsagePercent' "$HOME/.claude/context.json" 2>/dev/null)
echo "${value:-0}% | context"

これで動いた：
jqインストール後、ステータスバーが正常にコンテキスト使用率を表示するようになりました。 cygpathの有無も、PowerShellの文字コードも、関係なくなります。

5. まとめ：最初にやるべきこと

Windows環境でClaude Codeのステータスラインを設定するなら、 最初に jq をインストールするだけで3時間の試行錯誤が不要になります。

Windows環境での推奨セットアップ手順

  1. winget install jqlang.jq を実行してjqをインストール
  2. ~/.claude/statusline-command.sh を作成（jqを使ったシェルスクリプト）
  3. ~/.claude/settings.json に "statusCommand" を追加
  4. Claude Codeを再起動して動作確認

日本語フォルダ名という地雷について

今回の問題の一因は、プロジェクトフォルダ名に日本語が含まれていたことです。 Windows上でBashスクリプトを動かすとき、日本語パスはShift-JISとUTF-8の混在問題を引き起こしやすい。

開発系のフォルダは英数字のみで命名するのが鉄則ですが、 既存のフォルダ名を変えられない事情があることもあります。 そういう場合は jq のように「Windowsのロケール設定を参照しないツール」で処理するのが安全です。

まとめると：
① Claude Code経由のPATHはGit BashのPATHとは別物 → cygpathが使えない
② PowerShellのGet-ContentはデフォルトでShift-JISを想定することがある → 日本語パスで文字化け
③ jqを最初に入れればどちらも回避できる
