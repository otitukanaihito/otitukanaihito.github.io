AI CLIツールのスキル管理術 — Claude Code × AntiGravity 両刀使いの実践記

SKILL.md 1つで動く手軽さと、複数ツール運用の現実的な落とし所

はじめに

AI CLIツールを使い始めると、しばらくして「スキル」という機能に行き着きます。 定型作業をテンプレート化して、AIに「あのスキルを使って」と頼むだけで動かせる仕組みです。

問題は、Claude CodeとAntiGravityのように複数のAI CLIツールを併用し始めたとき。 ツールごとにスキルの置き場所が違うので、「同じスキルを両方に入れるの？」と悩み始めます。

この記事は、その悩みにぶつかった私が実際にたどり着いた運用方法の記録です。 正解ではなく「今これでうまく回っている」という実践報告として読んでください。

この記事の対象読者：
Claude Code または AntiGravity を使っていて、スキル機能を使い始めたばかりの方。 複数ツールを併用していて、スキルの管理方法に迷っている方。

1. そもそも「スキル」とは何か

Claude Code のスキルは、SKILL.md というファイルを含むフォルダが1つの「スキル」になります。 それだけです。フォルダ構成はこんなイメージです： 

~/.claude/skills/
├── weekly-review/
│   └── SKILL.md      ← 週次振り返りのスキル
├── deploy-site/
│   └── SKILL.md      ← サイトデプロイのスキル
└── daily-report/
    └── SKILL.md      ← 日報作成のスキル

SKILL.md の中身は、AIへの指示書です。 「このスキルが呼ばれたら、こういう手順でこういう出力をしてください」と書いておくと、 Claude Codeがそれを読んで実行してくれます。

スキルで何ができるか

私が実際に使っているスキルの例を挙げると：

  • 週次振り返り：今週の作業ログを読み込んで、KPT形式でまとめる
  • サイトデプロイ：静的サイトの更新手順（ファイル確認 → コピー → sitemap更新）をステップごとに実行
  • 日報作成：送迎記録のCSVを読んで、決まった書式で日報を出力する

要するに「毎回同じことをAIに説明するのが面倒な作業」を一度 SKILL.md に書いておけば、 次回から「週次振り返りのスキルを使って」の一言で動きます。

SKILL.md の最小構成例

# 週次振り返りスキル

## 概要
今週の作業ログをもとにKPT形式の振り返りを作成する。

## 手順
1. `~/logs/weekly/` 配下の最新ファイルを読み込む
2. Keep / Problem / Try に分類してまとめる
3. 来週のアクション3件を提案する

## 出力形式
Markdown形式で出力すること。

2. 複数ツールを使うときの悩み

Claude Code と AntiGravity では、スキルの置き場所が異なります。

ツールごとのスキルディレクトリ：
Claude Code → ~/.claude/skills/（グローバル、ユーザーホーム以下）
AntiGravity → .agent/skills/（プロジェクト内、ローカル）

「週次振り返り」スキルを両方で使いたい場合、同じ SKILL.md を2か所に置く必要が出てきます。 片方を変更したらもう片方も変更しなければならない。この二重管理がじわじわ面倒になってきます。

「シンボリックリンクで一元管理すればいい」という解決策が頭に浮かびます。 実際それは正しいのですが、Windowsではシンボリックリンクに権限が必要だったり、 Git管理下でシンボリックリンクが意図しない挙動をすることもあります。 最初からそこを目指すと、スキルを1つも作らないまま設計に時間を使い果たしてしまいます。

3. 実際にやった解決策

方針：シンプルに始めて、必要になったら整理する

現時点での私の運用はこうなっています：

  1. 共通スキルは両方に同じ SKILL.md を配置する（コピーで管理）
  2. ツール固有のスキルは片方だけに置く
  3. シンボリックリンクは「本当に必要になったとき」に検討する

共通スキルは、変更頻度が低い「定型作業の手順書」に限定しています。 週に1回変更するようなスキルは、どちらかのツールに寄せて「そっちでしか使わない」と割り切る。 これで二重管理の痛みはかなり減りました。

ツール固有のスキルは分ける

AntiGravity には // turbo というアノテーションがあり、 特定のサブエージェント（browser_subagent など）を呼び出すスキルを書けます。 これは AntiGravity 専用の記法なので、そのまま Claude Code に移植しても動きません。

逆に Claude Code 固有の仕組み（例：// claude_code_specific 的な記法や 特定のツール呼び出し）を使うスキルも、無理に AntiGravity に持ち込まないほうが安全です。

スキルの分類フローチャート

  • → ツール固有の記法・機能を使っている？ → 片方だけに置く
  • → 変更頻度が高い（週1以上）？ → メインで使うツールにだけ置く
  • → どちらでも使う、かつ変更頻度が低い？ → 両方にコピーして管理

4. ワークスペース構成のポイント

1フォルダにまとめるメリット

私はナレッジ・日記・複数プロジェクトを1つのフォルダ以下にまとめて管理しています。 AIツールから見ると「このフォルダを起点にすれば全部見える」状態になるので、 横断的な作業（「先週の日記を参照しながら振り返りを作る」など）がやりやすい。

注意点：git の誤操作で個人情報を公開しない

ナレッジや日記を含むフォルダを git 管理すると、 うっかり push したときに個人情報が公開されるリスクがあります。 私はこのフォルダに git init をしないことをルールにしています。 個別プロジェクトだけをそれぞれ独立した git リポジトリとして管理する方針です。

ワークスペースに git init しない理由：
日記・メモ・業務記録が混在するフォルダをうっかり GitHub に push すると、 個人情報・利用者情報の漏洩につながる可能性があります。 「プロジェクトが大きくなったら独立させる」を判断基準に、リスクを最小化します。

プロジェクトが大きくなったら独立させる判断基準

次のどれかに当てはまったら、独立したリポジトリに切り出すタイミングです：

  • 他の人と共同作業したくなった
  • CI/CD（自動デプロイ）を組みたくなった
  • 公開したいが、そのフォルダの他のファイルは公開したくない

逆に言えば、それ以外の段階では1フォルダにまとめておくほうが、 AI ツールとの相性がいいです。

5. AIの回答を出発点にする、正解にしない

この記事を書く元になった体験として、ひとつ正直に書いておきます。

スキルの管理方法を調べているとき、AIに「シンボリックリンクでスキルを共有する方法を教えて」と聞いたら、 ln -s コマンドを案内されました。Windowsで作業しているのに。

また、「AntiGravity のスキルをインストールするコマンドは？」と聞いたら、 存在しないコマンドを自信満々に教えてくれたこともあります。 npx antigravity-awesome-skills install みたいな、 実際には動かないものを。

AIの回答との正しい付き合い方：
AIが教えてくれることは「出発点」であり「正解」ではありません。 特にコマンドやパスに関する情報は、実際に動かして確認するまで信用しないこと。 「なぜそのコマンドが存在するのか」「自分の環境（OS・シェル）に合っているか」を 自分で考える癖をつけることが、AI時代の基礎リテラシーだと思います。

この問題はAIが悪いのではなく、使い方の問題です。 「AIに聞いたから正しい」ではなく、「AIに聞いてアイデアをもらって、自分で検証する」。 そのサイクルが確立できると、AIツールは本当に強力な武器になります。

6. まとめ：AI CLIツールとの付き合い方

この記事のポイントまとめ

  • 1. スキルは SKILL.md 1つのフォルダ。最初はシンプルに始める
  • 2. 複数ツールのスキル管理は「共通はコピー、固有は片方だけ」でまず運用する
  • 3. シンボリックリンクは「本当に必要になったとき」に検討すれば十分
  • 4. ワークスペースに git init しない（個人情報保護）
  • 5. AIの回答は出発点。自分の環境で検証して初めて「正解」になる

最初から完璧なスキル管理構成を目指さなくていいです。 使いながら育てていくのが、AI CLIツールとの正しい付き合い方だと実感しています。

スキルを自分で書くのが面倒なら、Claude Code に「このスキルを作って」と頼むのが最速です。 SKILL.md のたたき台を作ってもらって、そこから自分の用途に合わせて手直しするだけでOK。 AIツールは、AIツールを使うための作業にも使える。それがAI CLIの面白いところです。
