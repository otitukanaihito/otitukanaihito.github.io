GitHub PagesのLighthouseスコアをPerformance 65→77に改善した話

WebP変換・フォントpreload・lazy loading。小さなサイトでも効く5つの施策

はじめに

自分のブログ（このサイト）をChromeの開発者ツールで計測したとき、 Lighthouseの Performanceスコアが 65点 だと知って少しショックでした。

「静的サイトだし、そんなに遅くはないだろう」と思っていたのですが、 画像の最適化もフォントの読み込みもほぼデフォルトのまま。 改善できそうなことが明らかにたくさんありました。

この記事は、いくつかの施策を実施して 77点まで引き上げたときの記録です。 エンジニアでなくても取り組める内容ばかりなので、同じように静的サイトを運営している方の参考になれば。

この記事の対象読者：
GitHub Pages や静的サイトを運営していて、Lighthouseスコアを改善したい方。 フロントエンドの専門家でなくても取り組める施策に絞っています。

1. Lighthouseとは？なぜスコアが重要か

LighthouseはGoogleが開発した、ウェブページの品質を自動計測するツールです。 Chrome DevToolsから無料で使えて、Performance・Accessibility・Best Practices・SEOの4軸で評価してくれます。

Performanceスコアが重要な理由は主に2つです。

  • 検索順位への影響：Googleは「Core Web Vitals」をランキング要素として使用しており、表示速度が遅いと検索順位が下がる可能性がある
  • 直帰率への影響：ページの表示に3秒以上かかると、スマホユーザーの53%が離脱するというデータもある

Performanceスコアは以下の5つの指標の加重平均で計算されます：

指標 説明 配点
FCP 最初のコンテンツが描画されるまでの時間 10%
SI 画面が視覚的に埋まっていく速度 10%
LCP 最も大きな要素が描画されるまでの時間 25%
TBT メインスレッドがブロックされている合計時間 30%
CLS レイアウトのずれの大きさ 25%

2. 改善前の状態（65点）：何が問題だったか

Lighthouseのレポートで「Opportunities（改善の機会）」として挙げられていた主な問題は3つでした。

  1. 画像がPNG/JPGのまま：サイト内の画像がすべて従来形式で、ファイルサイズが大きい
  2. フォント読み込みがブロッキング：Google Fontsの読み込みが、初回描画を遅らせていた
  3. lazy loadingが未設定：スクロールしないと見えない画像も、ページ読み込み時に全部取得していた

見た目には大きな問題はなかったのですが、パフォーマンスの観点では「やれることを何もやっていない」状態でした。

3. やったこと5つ

施策1：WebP変換（LCP改善に効果大）

すべての画像をPNG/JPGからWebP形式に変換しました。 WebPはGoogleが開発した次世代画像フォーマットで、同等の品質でファイルサイズを 40〜60%削減できます。

Windowsでは「Squoosh」（無料ブラウザツール）を使って変換しました。 ドラッグ＆ドロップで変換できるので、エンジニアでなくても簡単に使えます。

HTMLの変更例

<!-- 変換前 -->
<img src="images/eyecatch.png" alt="アイキャッチ画像">

<!-- 変換後 -->
<img src="images/eyecatch.webp" alt="アイキャッチ画像"
     width="640" height="640"
     decoding="async">

施策2：画像にwidth/height属性を追加（CLS改善）

画像の縦横サイズをHTMLに明記することで、ブラウザが画像を読み込む前から「ここに640×640のスペースが必要」と判断できるようになります。 これにより、画像読み込み後にレイアウトがガタッとずれる現象（CLS）を防げます。

CLSはLighthouseスコアの25%を占める重要指標なので、ここの改善は効きました。

施策3：フォントのpreload化（FCP改善）

Google Fontsを使っている場合、デフォルトの読み込み方では描画がブロックされることがあります。 preload と font-display: swap を組み合わせることで、フォントが読み込まれる前に仮フォントで表示→後からちゃんとしたフォントに切り替える方式にしました。

フォント読み込みの最適化（head内）

<!-- preconnect で接続を先に確立 -->
<link rel="preconnect" href="https://fonts.googleapis.com">
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>

<!-- フォントCSSをpreloadして、onloadでstylesheetに切り替え -->
<link rel="preload" as="style"
    href="https://fonts.googleapis.com/css2?family=Inter:wght@400;600;700;800&display=swap"
    onload="this.onload=null;this.rel='stylesheet'">

<!-- JS無効環境向けフォールバック -->
<noscript>
    <link rel="stylesheet"
        href="https://fonts.googleapis.com/css2?family=Inter:wght@400;600;700;800&display=swap">
</noscript>

また、フォントウェイトを本当に使うものだけ（400・600・700・800）に絞って、不要なウェイトを読み込まないようにしました。 これだけでフォント関連の転送量がかなり減ります。

施策4：loading="lazy" の追加（SI改善）

ファーストビューに表示されない画像（スクロールしないと見えない位置の画像）に loading="lazy" を追加しました。 これにより、スクロールに近づいたタイミングで初めて画像の読み込みが始まるようになり、 初期ページ読み込み時の転送量が削減されます。

注意点：
ファーストビューの「メインビジュアル」や「ヒーロー画像」には loading="lazy" を付けてはいけません。 逆に遅くなります。LCPの対象になるような画像には rel="preload" で先読みするのが正解です。

施策5：Tailwind CSSをビルド済みファイルで配信（FCP改善）

最初はTailwind CSSのCDNスクリプトを使って、ブラウザ上でリアルタイムにCSSを生成していました。 これは「CDN版はJavaScriptでCSSを動的生成するため重い」という問題があります。

ビルドツールを使って「このサイトで実際に使っているクラスだけを含んだCSSファイル」を事前に生成し、 静的ファイルとして配信するように変更しました。 CDN版51KB → ビルド済み（使用分のみ）への削減でFCPが改善しました。

4. 改善後の結果（77点）

5つの施策を実施した後、Lighthouseで再計測したところ 65 → 77点 に改善されました。 特に効果が大きかったのはWebP変換（LCP）と width/height 追加（CLS）の2つでした。

改善ポイントまとめ

LCP改善

WebP変換で画像ファイルサイズ40〜60%削減。ページの最大要素が速く描画されるようになった。

CLS改善

width/height属性の追加で、レイアウトのズレがほぼゼロに。スコアへの影響が25%と大きい指標。

FCP改善

フォントpreload化とTailwindのビルド化で、最初のコンテンツ描画が速くなった。

SI改善

lazy loading で初期ロードの転送量を削減。下部のコンテンツ読み込みが邪魔しなくなった。

5. まだやれること（77→90点以上を目指すなら）

77点でも一応「オレンジ」圏内で、まだ改善の余地はあります。 次にやるとしたら以下の施策です。

施策 期待効果 難易度
GA4/GTMの遅延読み込み（ユーザー操作後に発火） TBT -200ms 中
Critical CSS のインライン化（Above-the-fold CSS） FCP -0.3s 中
画像の <picture> + srcset（レスポンシブ画像） LCP改善（モバイル） 低
フォントのサブセット化 FCP改善 低
Service Worker導入（キャッシュ戦略） リピーターのSI改善 高

今のところは77点で満足しています。 Googleのガイドラインでは「90点以上でGreen（良好）」ですが、 コンテンツの質や更新頻度に比べると優先度は低いかなと考えています。

6. まとめ：小さなサイトでもLighthouseは見る価値がある

この記事のポイントまとめ

  • 1. LighthouseはChrome DevToolsで無料計測できる。まず現状を知るところから
  • 2. WebP変換が一番コスパが高い。Squooshで変換→HTMLを書き換えるだけ
  • 3. width/height 属性の追加はCLSに効く。忘れがちだが効果大
  • 4. フォントは preload + display=swap の組み合わせが基本
  • 5. lazy loading はファーストビュー以外の画像にだけ付ける（ヒーロー画像には付けない）

「静的サイトだからパフォーマンスはいいはず」と思っていましたが、 計測してみると改善できることがたくさんありました。 Lighthouseは問題を指摘するだけでなく、「何をすればいいか」も具体的に教えてくれます。 まだ試したことがない方は、ぜひ一度計測してみてください。
